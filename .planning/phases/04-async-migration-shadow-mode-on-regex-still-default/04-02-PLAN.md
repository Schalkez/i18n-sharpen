<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → AST pool | Untrusted source-file *content* is parsed by the workspace TS/framework compiler (parser-only, `createSourceFile` — no execution, no `Program`/type-checker). |
| parseFile → detectUsedKeys | Per-file parse failures return as data (`FileParseError`, collected); only missing-compiler throws (fatal `I18nSharpenError`). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Denial of Service | AST pool parsing many/large files | mitigate | The bounded pool caps concurrent `parseFile` (and thus concurrent compiler memory/CPU) to `maxConcurrency` (default 4) — `runBoundedPool` guarantees at most N in flight. No `Promise.all` over all files. |
| T-04-05 | Information Disclosure | `log.warn` of `parseErrors` | mitigate | Warning format includes only `file:line: message` from `FileParseError` (compiler diagnostic), not raw file contents. D-14 carrier is a plain data interface, never an Error with a leaked stack. |
| T-04-06 | Denial of Service (availability) | one bad file aborting the scan | mitigate | Collect-and-continue (ERR-01/D-14): per-file errors are pushed to `parseErrors` and the pool continues; a file READ error degrades to empty source. Only missing-compiler intentionally aborts (fatal, by contract). |
| T-04-07 | Tampering | `useAst` reaching public config | accept→mitigate | Acceptance criteria grep-assert `useAst` absent from `src/types.ts` and zod schema; it lives only on internal `opts`/`options` params. |
</threat_model>
