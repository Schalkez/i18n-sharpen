# Milestone v0.3.0: Developer Experience

**Status:** ✅ SHIPPED 2026-05-30
**Phases:** 1-5
**Total Plans:** 13

## Overview

Sharpened the day-to-day developer loop around `prune`/`extract`/`validate` — key ordering, interactivity, smarter dynamic-key warnings, hardcoded-string detection, and a clean breaking-change release prep. Phase numbering was reset to 1 for this milestone (clean slate); originally 6 phases, reduced to 5 after Phase 1 scout discovered NSWRITE-01/02 already shipped in v0.2.x (commit `54712ab`).

## Phases

### Phase 1: Auto-Sorting Keys + Namespace Hardening

**Goal**: Users can opt in to deterministic key ordering on every locale file write performed by `extract` or `prune`, and the namespaced write-path is hardened (configurable default namespace, `--clean-empty` semantics, cross-file atomicity for prune).
**Depends on**: Nothing (NSWRITE-01/02 already shipped, composes on top)
**Requirements**: SORT-01..06, NSWRITE-03..05
**Plans**: 4 plans

Plans:
- [x] 01-01: Sort utility (`sortLocaleObject`) + `sortKeys` config + `--sort` CLI flag + getFiles determinism fix
- [x] 01-02: `defaultNamespace` config + replace hardcoded `"default"` + D-08 migration warning
- [x] 01-03: `--clean-empty` flag + `cleanEmptyNamespaceFiles` helper
- [x] 01-04: `writeLocaleFilesAtomic` two-phase commit + wire into extract/prune

### Phase 2: Dynamic Key Warnings

**Goal**: Developers see clearly separated, actionable warnings for structured-concat dynamic keys (with static prefix shown) vs fully-dynamic keys that cannot be analyzed — neither class pollutes the missing-key failure count.
**Depends on**: Nothing (additive to scanner + validator)
**Requirements**: DKEY-01..05
**Plans**: 3 plans

Plans:
- [x] 02-01: Classifier core (`dynamic.ts`) + types + zod schema + table-driven tests
- [x] 02-02: Validator integration + line-number tracking + `ignoreDynamicKeys` filter + grouped console summary + CHANGELOG
- [x] 02-03: Markdown report Dynamic Keys section + end-to-end integration tests

### Phase 3: Interactive Pruning

**Goal**: Developers running `prune` in a terminal can select exactly which unused keys to delete via an arrow-key TUI instead of accepting or rejecting all candidates at once.
**Depends on**: Nothing (additive flag on existing `prune`)
**Requirements**: IPRUNE-01..06
**Plans**: 3 plans

Plans:
- [x] 03-01: `PruneOptions.interactive` surface + CLI `--interactive` flag wiring
- [x] 03-02: Hand-rolled raw-mode TUI `runInteractivePrune` + unit tests + property test
- [x] 03-03: Integration into prune.ts with TTY detection, non-TTY fallback (D-14/D-15), summary preamble, CHANGELOG
- [x] Hardening (D-19/D-20): row truncation + `~` indicator, resize listener, injectable escDelay, split/Alt/double-Esc handling, viewport-height guard fallback

### Phase 4: Hardcoded String Detection

**Goal**: Developers can run `validate --check-hardcoded` to get a per-file report of raw text nodes inside HTML/JSX/Vue/Svelte/Astro tags that are not wrapped in a translation call.
**Depends on**: Phase 2 (shares scanner internals and validation report structure)
**Requirements**: HSTR-01..06
**Plans**: 2 plans

Plans:
- [x] 04-01: Parser and types implementation (hand-rolled template/JSX text-node scanner)
- [x] 04-02: CLI flag integration, terminal output, Markdown report, integration tests

### Phase 5: Deprecation Cleanup

**Goal**: The `I18nCopConfig` deprecated alias is fully removed from the public API and all internal references, completing the migration announced in v0.2.0 and preparing the codebase for a clean v0.3.0 release.
**Depends on**: Phases 1-4 (breaking change goes in final release prep)
**Requirements**: CLEANUP-01..03
**Plans**: 1 plan

Plans:
- [x] 05-01: Remove deprecated `I18nCopConfig` alias from types and index exports

## Milestone Summary

**Key Decisions:**
- Phase numbering reset to 1 for this milestone (prior milestone ended at Phase 10) — clean slate per milestone.
- Phase 1 originally split NSWRITE/SORT; merged after scout discovered NSWRITE-01/02 already shipped (commit `54712ab`). Remaining NSWRITE-03/04/05 folded into SORT phase (same locale write path).
- Interactive pruning implemented as a hand-rolled raw-mode TUI (no `inquirer`/`prompts` dep) to honor the tiny-dependency constraint.
- Cross-file prune atomicity implemented via two-phase commit (`writeLocaleFilesAtomic`).

**Issues Resolved:**
- Namespaced extract/prune write-routing gap (NSWRITE-01/02) — confirmed already closed by commit `54712ab`; the v0.2.0 PHASE-EXECUTION-REPORT had listed it as a known gap because it predated that commit.
- `I18nCopConfig` deprecated alias finally removed (last standing gap from v0.2.0).

**Technical Debt Incurred:**
- **Hardcoded-string detection (Phase 4) uses a hand-rolled regex/state-machine scanner** (`hardcoded.ts`), which accumulates edge-case patches (e.g. `<Foo.Bar>` tag names, JSX-in-expression boundaries). This is the primary motivation for the v0.4.0 AST-parser rewrite.
- Scanner key/attr/dynamic detection still regex-based across `regex.ts` / `dynamic.ts` — same AST migration target.
- Planning bookkeeping for this milestone was partially inconsistent at archive time (Phase 2 & 4 SUMMARY.md absent, Phase 5 had no phase directory) — work itself shipped and tagged `v0.3.0`.
